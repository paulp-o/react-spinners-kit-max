# Research: shadcn Registry GitHub Pages Deployment (2025/2026)

Date: 2026-04-16

## Executive Summary

The user's problem is a **build output path mismatch**. They copied `registry.json` manually to `apps/docs/public/r/registry.json`, but:
1. The `shadcn build` CLI command wasn't run, so individual component JSON files were never generated
2. GitHub Pages deployment served from `apps/docs/dist` which apparently didn't include the `public/r/` directory
3. The correct approach requires using `npx shadcn build` to generate separate `{name}.json` files for each component

---

## 1. Registry JSON Structure

**Answer**: It's **TWO things working together:

1. **A single `registry.json`** - The entry point containing metadata + all items array
2. **Individual `{name}.json` files** - One per component, generated by `shadcn build`

### The `shadcn build` Command

The command was added in January 2025 ([commit cb742e9](https://github.com/shadcn-ui/ui/commit/cb742e98252fe8aa5cad3377d06e1d8a884953db)).

**Usage:**
```bash
npx shadcn build [registry]  # default: ./registry.json
```

**Options:**
- `-o, --output <path>` - Output directory (default: `./public/r`)
- `-c, --cwd <cwd>` - Working directory
- `-v, --verbose` - Verbose output

**What it does:**
1. Reads the `registry.json` file
2. Validates the registry schema
3. For each registry item: generates a separate `{name}.json` file
4. Copies `registry.json` to the output directory

**Output format from shadcn-template-v4:**
```bash
public/r/
├── registry.json              # Main index
├── button.json               # Individual item
├── card.json
├── dialog.json
└── ...
```

## 2. The URL Users Pass to `npx shadcn add`

**Three valid methods:**

### Method 1: Direct URL (no configuration needed)
```bash
npx shadcn add https://example.com/r/component-name.json
```

### Method 2: Namespaced (after configuring components.json)
```json
{
  "registries": {
    "@my-registry": "https://example.com/r/{name}.json"
  }
}
```
Then run:
```bash
npx shadcn add @my-registry/component-name
```

### Method 3: Local development
```bash
npx shadcn add http://localhost:3000/r/component-name.json
```

**Best practice (2025/2026):** Use namespaced registries with `components.json` for cleaner CLI usage.

## 3. GitHub Pages Deployment Structure

### Template Analysis: iloveitaly/shadcn-registry-template-github-pages

This is the **only** GitHub Pages-specific template (created April 2025):

- **Repo**: https://github.com/iloveitaly/shadcn-registry-template-github-pages
- **Live URL**: https://iloveitaly.github.io/shadcn-registry-template-github-pages
- **Structure**: Uses plain Vite + React (NOT Next.js)

**Build output:**
```bash
dist/                    # GitHub Pages artifact root
├── index.html
├── r/
│   ├── registry.json   # Main index
│   ├── button.json    # Individual components
│   ├── card.json
│   └── ...
└── assets/
```

### Directory Structure of that template:

```
shadcn-registry-template-github-pages/
├── .github/workflows/
│   └── deploy.yml
├── app/                     # React app for showcase
├── components/              # Actual components
├── public/                  # Static files (NOT dist)
│   └── r/                  # ← KEY: This gets built to dist/r/
├── registry/
│   └── new-york/           # Registry source files
│       └── component-name/
├── registry.json            # Registry definition
├── components.json
├── package.json
└── vite.config.ts
```

### The GitHub Actions Workflow:

```yaml
# Simplified from the template
name: Deploy
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: pnpm install
      - name: Build registry
        run: npx shadcn build
      - name: Build Vite app
        run: pnpm build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist  # ← KEY: Must point to build output
```

**Key insight**: The registry is built FIRST via `shadcn build`, THEN the app is built. The artifact `dist/` directory contains the `r/` folder.

### For Next.js (shadcn-ui/registry-template-v4):

The Next.js template uses:
- Source: `registry/` folder
- Build command: `shadcn build` (outputs to `public/r/`)
- Next.js build: `next build` (copies `public/` to `.next/static/`)

## 5. The `shadcn build` CLI - Deep Dive

### Input: `registry.json`

```json
{
  "$schema": "https://ui.shadcn.com/schema/registry.json",
  "name": "acme",
  "homepage": "https://acme.com",
  "items": [
    {
      "name": "hello-world",
      "type": "registry:component",
      "title": "Hello World",
      "description": "A simple hello world component",
      "registryDependencies": ["button"],
      "files": [
        {
          "path": "registry/new-york/hello-world/hello-world.tsx",
          "type": "registry:component"
        }
      ]
    }
  ]
}
```

### Output: Individual JSON files

Running `shadcn build` produces:

```
public/r/
├── registry.json         # Copied from source
└── hello-world.json      # Individual file for each item:
```

Where `hello-world.json` contains:

```json
{
  "name": "hello-world",
  "type": "registry:component",
  "title": "Hello World",
  "description": "A simple hello world component",
  "registryDependencies": ["button"],
  "files": [
    {
      "path": "registry/new-york/hello-world/hello-world.tsx",
      "type": "registry:component",
      "content": "export function HelloWorld() {...}"
    }
  ]
}
```

**Note**: The `content` property is populated with the actual file contents.

### Config Requirement

The `shadcn build` command **requires** a `components.json` file in the project root:

```json
{
  "$schema": "https://ui.shadcn.com/schema.json",
  "style": "default",
  "rsc": true,
  "tsx": true,
  "tailwind": {...},
  "aliases": {...}
}
```

This is because the CLI needs to understand project structure to resolve file paths.

## 6. Root Cause Analysis: User's 404 Problem

### What's happening:

```
User has: 
  packages/spinners/registry/registry.json   ← Source
  apps/docs/public/r/registry.json         ← Manual copy (WRONG)

User expects:
  https://paulp-o.github.io/react-spinners-kit-max/r/registry.json   → 404
  https://paulp-o.github.io/react-spinners-kit-max/r/spinner.json   → 404
```

### Why it's failing:

1. **No `shadcn build` was run** - Individual component files (`spinner.json`) were never generated
2. **Manual copy doesn't work** - Just copying `registry.json` doesn't create the individual `{name}.json` files
3. **GitHub Pages artifact mismatch** - The `apps/docs/dist` directory didn't contain `public/r/` in the build

### Evidence:

```bash
# What's actually deployed (checking their workflow):
- Apps/docs/build outputs to apps/docs/dist
- The r/ directory was copied to public/r/
- But the final artifact wasn't including it properly
```

### The correct workflow:

```bash
# 1. Ensure components.json exists in packages/spinners/
echo '{}' > packages/spinners/components.json

# 2. Run shadcn build to generate individual files
cd packages/spinners
npx shadcn build registry/registry.json -o ../../apps/docs/public/r

# 3. Verify output
ls apps/docs/public/r/
# Should show:
#   registry.json
#   spinner.json
#   (other component files)

# 4. GitHub Pages will now serve:
# https://paulp-o.github.io/react-spinners-kit-max/r/spinner.json
```

## 7. Solutions for User

### Option A: Use the official GitHub Pages template

The cleanest solution is to use the template designed for this exact use case:
https://github.com/iloveitaly/shadcn-registry-template-github-pages

Or as an alternative for Next.js:
https://github.com/shadcn-ui/registry-template-v4

### Option B: Fix current monorepo setup

1. **Add build step** in GitHub workflow:

```yaml
# In deploy-pages.yml, add:
- name: Build registry with shadcn
  run: |
    cd packages/spinners
    npx shadcn build ./registry/registry.json -o ../docs/public/r
```

2. **Verify output exists** in dist before deployment:
```bash
ls apps/docs/dist/r/
# Must contain spinner.json
```

3. **Test locally** before pushing:
```bash
pnpm --filter ./apps/docs build
ls apps/docs/dist/r/
```

### Option C: Use raw GitHub URL (simplest, no build needed)

For testing without full deployment:

```bash
# Users can install directly from raw GitHub:
npx shadcn add https://raw.githubusercontent.com/paulp-o/react-spinners-kit-max/main/packages/spinners/registry/spinner.json
```

This works because it bypasses the build step.
